# Ответы на вопросы

## Аутентификация

В большинстве проектов, где я работал, JWT был стандартом де-факто — особенно в микросервисных архитектурах, где нет единого места для хранения сессий. Но был один проект — внутренняя аналитическая платформа для одного из подразделений ВПК (не могу называть детали по NDA) — где мы намеренно ушли от JWT в пользу сессионных токенов на Redis. Причина была прозаичная: система работала в изолированном контуре, все сервисы за одним reverse proxy, и передавать 300-500 байт токена в каждом запросе из браузера (а запросов было много — real-time дашборды) создавало заметный overhead. Переход на короткий session id в cookie + Redis lookup дал ощутимое снижение трафика и упростил инвалидацию сессий при выходе пользователя — с JWT это всегда боль, нужно либо blacklist, либо мириться с тем, что токен живёт до expiry.

Выбрал бы я JWT сейчас? Да, в большинстве случаев — особенно если сервисов несколько и они независимы. Stateless удобен при горизонтальном масштабировании: не нужно думать о sticky sessions или репликации Redis. Но в проектах с одним монолитным бэкендом и высоким RPS я бы серьёзно рассмотрел httpOnly cookie + server-side session. Ещё один момент: в проекте на WOW Land, где был игровой бэкенд на Node.js с WebSocket-ами, JWT в handshake работал хорошо — один раз проверили при подключении, дальше держишь user context в памяти. Там JWT был оправдан полностью.

## Идентификация

UUID — удобная штука, пока не начинаешь думать про B-tree индексы в PostgreSQL. Столкнулся с этим на проекте в ВятГУ, где делали систему учёта студентих работ. Таблица submissions росла быстро — около 50к записей в семестр, с историей за несколько лет. UUID v4 как primary key давал фрагментацию индекса: каждая новая запись вставлялась в случайное место B-tree, что при активной записи приводило к заметному degradation на INSERT. На PostgreSQL 14 это было не критично, но когда добавили partitioning и начали VACUUM агрессивнее, стало видно в explain analyze — bloat на индексе был раза в три больше, чем с serial.

Решили частично — перешли на UUID v7 (time-ordered), что сильно улучшило ситуацию. Но момент был показательный: UUID как "универсальный стандарт" работает отлично для распределённой генерации без координации, но в нагруженных таблицах с sequential insert-паттерном стоит думать либо о v7/ULID, либо о composite key где первым идёт timestamp.

Ещё один кейс — URL-ы в REST API. UUID в URL длинный и некрасивый, пользователи иногда копируют ссылки вручную. Мелочь, но в B2C проектах это заметно.

## Организация кода

NestJS — это первый фреймворк на Node.js, где DI и модульность реализованы настолько близко к тому, как это делается в Spring или .NET. Когда я впервые его попробовал после Express-а, первая реакция была "зачем столько абстракций". Но после нескольких месяцев работы в команде понял реальную пользу.

Конкретный пример: на одном из проектов (B2B SaaS, несколько команд) мы использовали NestJS модули как границы ответственности между командами. Каждая команда владела своим модулем, экспортировала только то, что нужно снаружи. Это реально снизило количество "случайных зависимостей" — когда разработчик импортирует что-то из чужого сервиса напрямую, обходя публичный интерфейс. DI делал это невозможным без явного экспорта.

Middleware и guards тоже окупились: AuthGuard вынесли один раз, и все контроллеры используют его через декоратор — без копипасты проверки токена в каждом хендлере.

Что создавало сложности — circular dependencies. В нескольких местах сервисы ссылались друг на друга, и NestJS падал с не самой информативной ошибкой. Решали через forwardRef(), но это ощущается как костыль. Ещё — декораторы TypeScript иногда работают неочевидно с наследованием классов, что давало странные баги при попытке расширить базовый сервис.

## Реактивность

React отлично работает для большинства UI-задач, но я сталкивался с ситуациями, где реактивная модель была скорее обузой.

На проекте в ВятГУ делали визуализацию для учебного симулятора — canvas-based рендер с физикой, ~60fps, сотни объектов. Попытка интегрировать это с React state была катастрофой: каждый tick обновлял состояние, React перерисовывал дерево компонентов, даже мемоизированных, что давало заметные просадки FPS. В итоге вынесли всю игровую логику полностью за пределы React, рендерили напрямую в canvas через requestAnimationFrame, а React использовали только для HUD (счёт, кнопки паузы). Это дало x3 прирост производительности — никаких reconciliation overhead в hot path.

Конкретные неудобства в React: useEffect с зависимостями — это постоянный источник багов. Либо забываешь что-то добавить в deps array и ловишь stale closure, либо добавляешь лишнее и получаешь infinite loop. React 18 с concurrent features немного улучшил ситуацию, но концептуально useEffect всё равно остаётся самой непрозрачной частью хука-модели.

Ещё неудобно: когда компонент должен реагировать на внешнее событие (WebSocket message, gamepad input), приходится либо городить useRef + forceUpdate, либо тащить state management библиотеку. В Vue 3 с Composition API это решается чище через reactive() без лишних оберток.
