# Ответы на вопросы

## Аутентификация

В проекте для оборонки мы используем сессионную аутентификацию через Redis. Каждый запрос несёт только короткий session ID (32 байта), а не полный JWT-токен с payload'ом. Валидация — одно обращение в Redis, которое и так дёшево при локальном деплое. Выбрали это решение сознательно: у нас закрытая сеть, несколько инстансов бэкенда за одним reverse proxy, и Redis уже стоял под кэш — добавить session store обошлось дёшево.

JWT я бы выбрал там, где реально нужна stateless-архитектура: например, если бэкенд горизонтально масштабируется и хранить сессии в Redis накладно по деньгам или сложности. В WOW Land мы делали публичный API для игрового лобби — там JWT был оправдан, потому что не было централизованного хранилища состояния.

Против JWT я бы выступил в двух сценариях. Первый — когда нужна мгновенная инвалидация токена (например, пользователя забанили или он сменил пароль). С JWT без blacklist это в лучшем случае "подождать пока не истечёт TTL". Второй — когда payload раздувается: в одном проекте мы запихали в JWT роли, permissions и организацию, и каждый запрос гнал 2 KB туда и обратно. В итоге перешли на opaque token + кэш ролей на сервере.

## Идентификация

Проблема с UUID была в геймдевном проекте на Unity/WOW Land — там мы использовали UUID как первичный ключ для игровых объектов и потом делали сортировку по id для пагинации в лидерборде. UUID v4 — случайный, порядок непредсказуем, индекс фрагментируется жёстко при вставке. При 50k записей запросы с ORDER BY id + LIMIT стали заметно медленнее, чем если бы ключ был monotonically increasing (serial или UUID v7).

В итоге для лидерборда добавили отдельное поле `created_at` с индексом и сортировали по нему. Но неприятный осадок остался: выбери мы сразу UUID v7 (time-ordered) или ulid — проблемы бы не было.

Второй кейс — внешние системы. В проекте интеграции с одним legacy API нам передавали UUID как строку, но их система ожидала UUID без дефисов (32 hex-символа). Пришлось добавлять слой трансформации повсюду.

## Организация кода

NestJS я впервые применил серьёзно именно в бэкенд-части для WOW Land. DI сначала казался overhead'ом — зачем, если можно просто `new Service()`? Польза проявилась, когда нужно было заменить реальный EmailService на MockEmailService в тестах. Одна строка в тестовом модуле — и всё. Без DI пришлось бы мокать на уровне модуля или передавать зависимости руками.

Guards и декораторы реально сэкономили код в проекте с ролями. Вместо того чтобы в каждом контроллере писать `if (req.user.role !== 'admin') throw new ForbiddenException()`, повесили `@Roles('admin')` + `RolesGuard` — и забыли. Это не просто DRY, это невозможность случайно забыть проверку.

Что создавало сложности — circular dependencies между модулями. Когда GamesModule импортирует AuthModule, а AuthModule нужен GamesModule для проверки токена — NestJS валится с не очень информативной ошибкой. `forwardRef()` спасает, но это сигнал что архитектура немного съехала. В большом проекте с 15+ модулями такие circular deps начинают встречаться и требуют рефакторинга.

## Реактивность

Был конкретный случай в этом же проекте с игровым лобби: у нас было дерево компонентов с глубиной ~6 уровней, и корневое состояние хранило список игроков онлайн (обновлялся через polling каждые 2 секунды). При каждом обновлении перерисовывалась вся ветка, включая карточки игроков которые не изменились. Добавили `React.memo` и `useMemo` для дорогих вычислений — стало лучше, но код стал грязнее. Реактивность создала проблему которую мы потом фиксили оптимизациями.

Второй неудобный кейс — canvas-анимация. В Unity и в нескольких веб-проектах мы рендерили через requestAnimationFrame напрямую. Там логика "я сам решаю что и когда рисовать" — быстро и предсказуемо. Когда пытаешься сделать то же самое в React, useEffect + useRef + dependency array превращается в танец с бубном, чтобы анимация не перезапускалась при каждом ре-рендере.

Что конкретно раздражает в React: dependency array в useEffect ощущается как ручное управление памятью — eslint-plugin-react-hooks спасает, но не всегда. И то что `useEffect` с async функцией требует оборачивания через IIFE или отдельную функцию — minor, но постоянное friction.
